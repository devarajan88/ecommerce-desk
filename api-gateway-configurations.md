# API Gateway Configurations

This document outlines the Load Balancer, Rate Limiter, and resilience configurations implemented in the API Gateway of the Saga Pattern Spring Boot Demo project. These configurations act as global filters and policies applied across all underlying microservices (e.g., user-service, orders-service, products-service, payments-service).

## 1. Dependency Adjustments

To support the reactive nature of the Spring Cloud Gateway's rate limiter, the Redis dependency in `api-gateway/pom.xml` was updated:

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-redis-reactive</artifactId>
</dependency>
```

This ensures full compatibility with the non-blocking `RequestRateLimiter` Gateway filter.

## 2. Rate Limiting Strategy
A global rate limiting rule is designed to restrict rapid-fire API hits. 

**Key Resolver (`RateLimiterConfig.java`):**
A custom `KeyResolver` bean identifies users by their incoming IP Address.
```java
@Configuration
public class RateLimiterConfig {
    @Bean
    public KeyResolver userKeyResolver() {
        return exchange -> Mono.just(Objects.requireNonNull(exchange.getRequest().getRemoteAddress()).getAddress().getHostAddress());
    }
}
```

**Filter Configuration (`application.yml`):**
The `RequestRateLimiter` is added globally under `spring.cloud.gateway.default-filters`.
- `replenishRate`: **10** (number of requests that can be made per second)
- `burstCapacity`: **20** (maximum number of requests allowed in a burst)
- `requestedTokens`: **1** (cost of each request)

## 3. Resilience and Fallbacks

A central `FallbackController` handles situations when a microservice is unresponsive, preventing unhandled 500 server errors or infinite hanging requests from reaching the client.

```java
@RestController
@RequestMapping("/fallback")
public class FallbackController {
    @GetMapping
    public Mono<ResponseEntity<String>> fallback() {
        return Mono.just(ResponseEntity.status(HttpStatus.SERVICE_UNAVAILABLE)
                .body("Service is currently unavailable. Please try again later."));
    }
}
```

This controller handles traffic that matches the `CircuitBreaker` Gateway filter configured with `fallbackUri: forward:/fallback`.

## 4. Default Filter Setups (`application.yml`)

The following settings were configured in the API Gateway's `application.yml` to apply automatically to all routes.

### Gateway Default Filters

1. **RequestRateLimiter**: Uses Redis and `userKeyResolver` for limiting.
2. **Retry Filter**: 
   - Retries up to 3 times on `GET` requests for `SERVER_ERROR` or specific network exceptions like `java.io.IOException` or `TimeoutException`.
   - Incorporates an exponential backoff strategy starting at 50ms and maxing out at 500ms.
3. **CircuitBreaker Filter**: Routes failing, stalled, or timed-out requests to the generic Spring controller `forward:/fallback`.

### Global Network Settings

1. **HTTP Client Handling:**
   - Connect timeout: `2000` ms
   - Response timeout: `5` seconds
2. **Load Balancer (Spring Cloud LoadBalancer):**
   - Configured `spring.cloud.loadbalancer.use404: true` to clearly distinguish when an expected service no longer exists in Eureka registries.

### Resilience4j Granular Controls

Configured under the generic `resilience4j` YAML node:
- **Circuit Breaker defaults:** Sliding window size of 10, failure rate threshold of 50%, and an open-state wait duration of 10 seconds before attempting half-open calls (allowing max 5 permitted calls).
- **Time Limiter defaults:** Standardizes upstream call timeouts natively within 3 seconds, complementing connection drop logic.

## 5. Centralized Logging

A robust global filter (`LoggingFilter.java`) intercepts every request and response passing through the API Gateway, enabling comprehensive centralized logging and analytics:

- **Client IP Tracking**: Resolves and records the exact IP address of the incoming caller.
- **Request Metrics**: Logs the precise HTTP Method, URI path, and query parameters before routing to downstream microservices.
- **Execution Time Parsing**: Calculates dynamically the end-to-end execution duration (in milliseconds) spent across internal services.
- **Status Codes Monitoring**: Logs the exact HTTP response code generated by the entire chain.

Furthermore, integrating with **Zipkin/Micrometer Tracing** automatically injects `Trace-ID` and `Span-ID` into every logging statement via MDC (Mapped Diagnostic Context). This ensures requests logged in the Gateway can be trivially correlated to logs within `user-service`, `orders-service`, etc., in any aggregator (ELK, Grafana Loki).

```java
// Example Logging Output
// INFO [api-gateway, 5f3d4a67b..., 5f3d4a67b] Incoming Request => IP: 192.168.1.1, Method: GET, Path: /users/123
// INFO [api-gateway, 5f3d4a67b..., 5f3d4a67b] Outgoing Response => Path: /users/123, Status Code: 200 OK, Execution Time: 45 ms
```

---
These configurations ensure that the API Gateway acts as a highly resilient, throttling, and load-aware proxy, efficiently protecting and tracking all backend microservices from unusual loads or single points of failure.
